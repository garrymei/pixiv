# 粤次元君 (Yue Ci Yuan Jun)

## 一、项目概述
“粤次元君”是一款专为粤港澳大湾区二次元爱好者打造的社交创作社区。平台致力于连接大湾区内的 Coser、摄影师、妆娘、毛娘、道具师及广大二次元同好，提供一个集作品展示、日常交流、线下活动资讯及创作协作需求发布于一体的综合性平台。

## 二、产品定位
**大湾区二次元社交创作社区**
区别于泛娱乐社交平台，本项目垂直聚焦于二次元文化与线下创作协作。不仅提供内容消费（看图/看动态），更强调创作者之间的真实连接与互助（如：约拍、求妆、拼团、接单）。

## 三、目标用户
- **内容创作者**：Coser、二次元摄影师、画师、视频UP主。
- **专业协助者**：妆娘（化妆师）、毛娘（假发造型师）、道具师、后期师。
- **展会及活动方**：各大漫展主办方、同人展、Live演出组织者。
- **二次元爱好者**：热衷于看展、追番、购买谷子的普通同好。

## 四、MVP 范围 (Minimum Viable Product)
MVP 阶段旨在验证核心的“内容流转”与“需求撮合”闭环，包含以下核心功能：
1. **基础账户体系**：简单的注册、登录与个人资料设置。
2. **内容分享**：支持图文格式的作品分享与日常动态发布。
3. **活动中心**：官方活动资讯的展示与简单的报名参与功能。
4. **协作需求大厅**：支持发布与接单参与“摄影/模特/化妆/毛发/道具”的合作需求。

## 五、当前阶段不做什么（反向边界）
为保证项目不跑偏，在 **第 0 阶段及 MVP 阶段**，坚决**不做**以下内容：
- ❌ 不写完整前端业务页面（仅做结构设计）。
- ❌ 不写完整后端业务逻辑（仅做骨架与 API 规范）。
- ❌ 不接微信登录正式流程（使用 Mock 登录或测试账号）。
- ❌ 不做复杂 UI 还原（采用基础组件库搭建骨架屏）。
- ❌ 不做推荐算法（内容展示仅按时间倒序或简单分类）。
- ❌ 不做私信功能（MVP 阶段可通过互换联系方式解决）。
- ❌ 不做消息推送（App Push / 微信订阅消息）。
- ❌ 不做完整后台管理系统（仅提供必要的数据查改 API或直连DB操作）。

## 六、核心模块
1. **用户模块 (User)**：鉴权、个人主页、关注关系。
2. **内容模块 (Content)**：作品集 (Works)、日常动态 (Daily)。
3. **活动模块 (Event)**：漫展/活动列表、活动详情、报名信息。
4. **需求协作模块 (Collab)**：需求广场（约拍/求妆/求毛等）、应征机制。

## 七、页面结构 (前端视图草案)
- **首页 (Home)**
  - 推荐Feed流（作品/日常动态）
  - 分区切换（作品区、日常区）
- **活动 (Events)**
  - 官方/展会活动列表
  - 活动详情页及报名入口
- **需求 (Collab/Market)**
  - 需求分类筛选（摄影、妆娘、毛娘、道具、模特）
  - 需求发布入口
  - 需求详情及参与按钮
- **我的 (Profile)**
  - 个人资料编辑
  - 我的发布（作品/动态/需求）
  - 我的报名（活动/需求）

## 八、数据实体 (MySQL 表设计初稿)
| 实体 | 描述 |
| --- | --- |
| `sys_user` | 用户主表（包含基本信息、角色认证） |

| `post_daily` | 日常动态表（类似朋友圈碎碎念） |

| `event_enroll` | 活动报名记录表 |
| `collab_req` | 协作需求表（约拍/求妆等） |
| `collab_apply` | 需求应征记录表 |
| `sys_department` | 组织架构表（当前库中已存在） |
| `relation_data_depart`| 数据与部门关联表（当前库中已存在） |

## 九、技术选型 (拟定)
- **前端**：UniApp (Vue3) - 兼顾微信小程序与多端打包 / 配合 UI 框架 (如 uView 或 Uni-ui)。
- **后端**：Python (FastAPI / Flask) 或 Node.js (待根据团队最终确认，目前验证过 Python 连通性)。
- **数据库**：MySQL (当前已配置并连通测试库 `demo-1gp5eo6c4e09685e`)。
- **缓存**：Redis（可选，MVP阶段视性能需求而定）。

## 十、开发阶段划分
1. **第 0 阶段 (当前)**：项目定向 (Project Alignment)。明确边界、输出文档与设计稿、完成基础设施联通验证。
2. **第 1 阶段**：环境搭建与 API 骨架开发。完成数据表创建及 Mock 接口。
3. **第 2 阶段**：MVP 核心业务逻辑开发与前后端联调。
4. **第 3 阶段**：内测体验与修复，核心功能验收。

## 十一、代码规范
1. **分支管理**：
   - `main`：生产环境主分支。
   - `develop`：开发环境主分支。
   - `feature/*`：功能特性分支。
2. **提交规范**：
   - 遵循 Angular Commit 规范，如 `feat: 增加用户登录接口`, `fix: 修复需求列表分页bug`, `docs: 更新 README`。
3. **编码风格**：
   - 前端：遵循 ESLint + Prettier。
   - 后端：遵循 PEP8（Python）或对应语言的主流规范。
