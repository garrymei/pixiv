# 粤次元君 (Yue Ci Yuan Jun)

## 一、项目概述

“粤次元君”是一款面向粤港澳大湾区二次元爱好者的社交创作社区小程序。

平台聚焦“内容创作 + 线下协作 + 活动连接”三大核心场景，连接：

- Coser
- 摄影师
- 妆娘 / 毛娘 / 道具师
- 二次元爱好者

构建一个既能看内容，又能约合作 / 找人 / 参与活动的垂直社区。

## 二、产品定位

🎯 大湾区二次元社交创作社区

区别于泛社交产品，本项目强调：

- 真实创作协作（约拍 / 求妆 / 拼团）

### 需求协作状态机（2026-04-22 更新）

以下规则用于“需求/技能发布”申请链路，前后端需严格一致：

1. 多人可申请，初始均为 `待确认`
- 发布后，多个用户可提交申请。
- 每条申请初始状态均为 `待确认`。

2. 发布方先“选人接受”，再进入确认阶段
- 当存在多个申请人时，发布方必须先从申请列表中明确选择 1 人点击“接受申请”。
- 被选中的申请记录进入 `accepted`（已接受，待对方最终确认）。
- 未被选中的申请记录保持原状态，不自动变已拒绝；活动时间过后统一转 `expired`（已过期）。
- 该需求一旦已有“被接受人”，不允许新的申请进入（接口返回 `already accepted`）。

3. 最终确认必须由申请方完成
- 发布方“接受申请”后，不等于最终生效。
- 只有“被发布方选中并接受的那一位申请人”可以执行“最终确认”。
- 其他未被选中的申请人不允许最终确认。
- 被接受的申请方完成“最终确认”后，双方才算达成约定。
- 达成后状态为 `已确认`，并按活动时间小时粒度自动流转：
  - 同小时：`进行中`
  - 过去小时：`已结束`

4. 双方确认后如需改发布信息，必须先“取掉约定”
- 在双方已确认约定期间，发布内容进入编辑锁定，不允许直接修改。
- 任一方可发起“取掉约定”，另一方必须确认。
- 只有双方都确认取消后，状态变为 `已取消约定`，发布方才可修改需求信息。

5. 过期规则
- 未被接受的申请：活动时间过后自动视为 `已过期`。
- 已接受但未完成最终确认的申请：活动时间过后同样为 `已过期`。

6. 当前状态枚举（展示文案）
- `pending_confirm`：待确认
- `accepted`：待对方最终确认
- `confirmed`：已确认
- `ongoing`：进行中
- `ended`：已结束
- `expired`：已过期
- `cancel_pending`：取消待确认
- `cancelled`：已取消约定

7. 当前关键接口动作
- `POST /demands/:id/apply`：申请需求（已接受时禁止）
- `GET /demands/:id/apply/status`：获取当前用户与该需求关系状态
- `GET /demands/:id/applications`：发布方查看申请列表并选择目标申请人
- `POST /demands/:id/apply/confirm`：发布方对指定 `application_id` 执行“接受申请” / 被接受申请人执行“最终确认”
- `POST /demands/:id/agreement/cancel/request`：发起取消约定
- `POST /demands/:id/agreement/cancel/confirm`：确认取消约定
- `PATCH /demands/:id`：作者修改需求（若存在已确认且未取消约定则拦截）

8. 记录模型说明
- 一条需求对应一个申请列表（`applications`），用于保留所有申请记录与状态轨迹。
- 需求本身仅允许维护一个当前有效的被接受关联（`accepted_application_id` / `accepted_user_id`）。
- 当双方完成取消约定后，该唯一关联会被清空，需求可重新进入可编辑和重新选择阶段。

## 十一、内容安全与审核（新增）

为避免违规内容直接出现在平台，项目对“用户上传头像、用户发布内容”引入了审核流与上传控制：

### 1. 审核策略

- 默认策略：发布/上传先走自动审核；自动判定可疑则进入待人工审核（`PENDING`），审核通过（`APPROVED`）后才会出现在公开列表中。
- 动态/需求：公开列表仅返回 `APPROVED` 内容；未审核通过内容不可通过详情接口直接访问。
- 头像：用户上传新头像后，若进入 `PENDING`，前台继续显示旧头像，新头像待审核通过后才生效。

### 2. 上传控制（必须）

上传接口 `/uploads/image` 已加入常规安全控制：

- 文件类型限制：仅允许 `.png/.jpg/.jpeg/.gif/.webp`。
- 文件大小限制：默认最大 5MB，可通过 `UPLOAD_MAX_BYTES` 配置（单位：字节）。
- 文件签名校验：上传落盘后会校验图片 magic header，防止伪装格式绕过。
- 简易频控：按用户维度限流，默认 `60s` 内最多 `20` 次上传，可通过 `UPLOAD_RATE_LIMIT_MAX` / `UPLOAD_RATE_LIMIT_WINDOW_MS` 调整。

### 3. 人工审核接口（管理员）

为避免审核接口暴露，人工审核接口需要管理员 Token：

- 配置：设置 `ADMIN_TOKEN` 环境变量
- 调用：在请求头中携带 `x-admin-token: <ADMIN_TOKEN>`

接口示例：

- `PATCH /admin/reviews/posts/:id` body：`{ "action": "approve" | "reject", "reason": "..." }`
- `PATCH /admin/reviews/demands/:id` body：同上
- `PATCH /admin/reviews/avatars/:userId` body：同上

## 十三、后台平台需求（新增）

当前项目已不再满足“只有小程序前台”的阶段，后续需要建设一个与小程序模块逐一对应的后台管理与监控平台。

建设目标：

- 每个小程序业务模块都有一个对应的后台管理入口
- 每个小程序业务模块都有一个对应的监控入口
- 后台既能做内容运营，也能做平台运维与异常排查
- 小程序端、后台 Web、后端 API 共用同一套业务数据模型

### 1. 后台平台设计原则

- 一个前台模块，对应一个后台模块
- 一个后台模块，至少包含：管理区、数据看板、异常监控、操作日志
- 管理能力与监控能力并行建设，不能只做内容录入页
- 优先围绕当前已上线模块建设，不空转做泛后台

### 2. 小程序模块 -> 后台平台映射

| 小程序模块 | 当前前台能力 | 后台管理能力 | 后台监控能力 |
| --- | --- | --- | --- |
| 首页 | Banner / 推荐流 / 活动入口 | 首页推荐位、Banner、排序规则、公告配置 | 首页接口成功率、首屏耗时、空数据告警、点击率 |
| 发现 | 动态广场、次元市集、标签筛选 | 内容管理、需求管理、标签管理、推荐词配置 | 搜索成功率、空结果率、列表空数据率、Mock 兜底次数 |
| 发布 | 发布帖子、发布需求 | 帖子审核、需求审核、敏感词、发布频控、失败日志 | 发帖成功率、需求发布成功率、上传失败率、慢请求排行 |
| 活动 | 活动列表、详情、报名、我的活动 | 活动 CRUD、是否可报名、报名名单、状态管理、核销签到 | 报名成功率、报名失败原因、满额告警、截止告警、活动转化率 |
| 我的/用户 | 资料编辑、我的帖子、我的需求、我的活动 | 用户档案、用户状态、内容总览、违规记录、封禁管理 | 资料修改成功率、异常账号识别、高频操作告警 |
| 登录会话 | 登录页、Mock 登录、Token 会话 | 登录记录、会话配置、账号策略、黑名单 | 登录成功率、401/403 统计、会话初始化失败率 |
| 上传资源 | 图片上传、封面读取、资源展示 | 上传文件管理、资源审查、失效资源清理 | 上传成功率、资源 404、上传耗时、非法资源告警 |
| 系统运维 | `/health`、域名、Nginx、后端服务 | 服务配置、告警中心、操作审计、运行配置 | 服务可用性、5xx 错误率、systemd 状态、证书到期提醒 |
