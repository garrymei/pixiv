# 粤次元君 - 首轮内测流程与反馈收集机制 V1

## 一、目标

首轮内测不能只是“大家随便看看”，而要让试用用户在明确时间段内，按明确任务去体验，并以统一格式提交反馈。

本方案的目标是：

- 让内测有组织、有节奏
- 让反馈可汇总、可比较、可复盘
- 让问题不只是散落在微信群和私聊聊天记录里

## 二、首轮内测原则

- 先小范围，优先找愿意认真体验的人
- 先跑核心链路，不做大而全探索
- 先收结构化反馈，再收自由建议
- 反馈必须留痕，不能只口头说

## 三、邀请方式

### 3.1 邀请对象建议

建议优先邀请以下 3 类人：

- 内容型用户：会发帖、会发需求的人
- 参与型用户：愿意报名活动、参与需求的人
- 普通观察型用户：从新用户视角看页面是否好懂、好看

### 3.2 邀请方式建议

推荐方式：

- 定向私聊邀请
- 小范围微信群 / 社群邀请
- 通过官方账号发布“首轮内测招募”

不建议：

- 一开始大范围公开扩散
- 没有任务说明就直接发链接让大家自己看

### 3.3 邀请文案建议

建议邀请文案至少说明：

- 本次是首轮内测
- 试用时间段
- 需要完成的任务
- 反馈提交方式
- 反馈价值：会直接影响后续版本优化

## 四、试用时间段建议

### 4.1 推荐周期

建议首轮内测周期控制在：

- 3~5 天

推荐节奏：

- 第 1 天：发邀请、分发任务、确认进入方式
- 第 2~4 天：集中试用与反馈提交
- 第 5 天：汇总问题、做首轮复盘

### 4.2 推荐时间安排

建议选择：

- 工作日晚间
- 周末下午 / 晚间

原因：

- 用户更容易完整走一遍体验任务
- 便于收集实时反馈

## 五、试用任务清单

### 5.1 核心任务建议

首轮内测建议每位用户至少完成以下任务中的 4~6 项。

| 编号 | 任务 | 目标 |
| --- | --- | --- |
| T-01 | 打开首页并浏览内容 | 看首页第一印象与内容密度 |
| T-02 | 点击一条帖子进入详情 | 看内容消费链路是否自然 |
| T-03 | 尝试发布一条帖子 | 看发布是否顺畅 |
| T-04 | 浏览活动页并进入活动详情 | 看活动信息是否清楚 |
| T-05 | 尝试报名 1 个官方活动 | 看报名链路是否顺 |
| T-06 | 浏览需求并进入详情 | 看需求内容是否易理解 |
| T-07 | 尝试参与 1 条需求 | 看参与动作是否顺 |
| T-08 | 进入“我的”并编辑资料 | 看个人中心链路是否顺 |

### 5.2 任务分组建议

如果用户类型不同，可分任务组：

- 内容型用户：重点做发帖、发需求、编辑资料
- 参与型用户：重点做浏览、报名、参与
- 普通新用户：重点看首页、活动页、页面理解度和整体感受

## 六、反馈提交通道

### 6.1 推荐收集方式

建议同时保留两类通道：

| 通道 | 用途 |
| --- | --- |
| 结构化反馈表单 | 用于收集标准化体验结果 |
| 群内 / 私聊补充 | 用于补充截图、录屏、追问上下文 |

### 6.2 推荐主通道

首轮建议以以下任一方式作为主通道：

- 腾讯文档表单
- 飞书表单
- 问卷星
- 简单在线表格

要求：

- 所有人统一走一个入口提交
- 不接受只在聊天里说“感觉不太好”

## 七、问题记录格式

每条问题建议统一记录以下字段：

| 字段 | 说明 |
| --- | --- |
| 问题编号 | 例如 `BETA-001` |
| 提交人 | 谁反馈的 |
| 时间 | 反馈时间 |
| 页面 / 模块 | 出问题的位置 |
| 操作场景 | 用户当时在做什么 |
| 问题描述 | 发生了什么 |
| 严重程度 | 高 / 中 / 低 |
| 是否可复现 | 可复现 / 偶发 |
| 截图 / 录屏 | 附件链接 |
| 建议处理方式 | 修复 / 优化 / 观察 |

## 八、建议反馈维度

首轮反馈建议至少覆盖以下维度：

| 维度 | 要问的问题 |
| --- | --- |
| 页面是否好看 | 页面风格是否有质感、是否像可用产品 |
| 功能是否看得懂 | 首页、活动页、需求页是否一眼知道怎么用 |
| 发布是否顺 | 发帖 / 发需求时是否卡住、是否有困惑 |
| 活动报名是否顺 | 报名入口、信息理解、完成反馈是否自然 |
| 是否愿意继续用 | 用户是否愿意再次打开 |
| 最想加什么功能 | 用户最明确的下一步期待是什么 |

## 九、首轮反馈建议结构

建议把反馈分成 3 层：

### 9.1 体验评分

- 页面是否好看
- 是否容易上手
- 是否愿意继续使用

### 9.2 链路问题

- 首页是否好理解
- 发布是否顺
- 报名是否顺
- 参与是否顺
- 编辑资料是否顺

### 9.3 开放建议

- 你最想加什么功能
- 你最不喜欢哪里
- 你最容易放弃使用的地方是什么

## 十、推荐执行流程

### 第 1 步：招募

- 确定首轮内测名单
- 发内测邀请说明
- 告知试用时间和任务

### 第 2 步：体验

- 用户按任务清单完成体验
- 遇到问题先截图 / 录屏

### 第 3 步：提交

- 统一填写反馈表单
- 补充截图或录屏链接

### 第 4 步：整理

- 按问题编号收敛到一张问题表
- 按页面、严重程度、频次分类

### 第 5 步：复盘

- 输出首轮复盘结论
- 进入需求池和缺陷池

## 十一、内测组织建议

### 建议角色分工

| 角色 | 职责 |
| --- | --- |
| 官方运营 | 发起邀请、说明任务、追收反馈 |
| 产品 / 负责人 | 整理问题优先级 |
| 开发 / 执行人 | 判断是否为缺陷、是否可复现 |
| 记录人 | 统一维护问题清单和反馈汇总 |

- [ ] 已有明确试用时间段
- [ ] 已有任务清单
