# 粤次元君 - 冷启动后迭代需求池管理方法 V1

## 一、目标

冷启动后会持续收到大量反馈，但“有人提”不等于“立刻开发”。

本方法的目标是：

- 让反馈先入池、再分类、再判断
- 避免团队被零散建议牵着走
- 让需求有统一记录框架和优先级标准

## 二、核心原则

- 先收集，不马上拍板开发
- 先判断是不是问题，再判断要不要做
- 优先处理阻碍使用的问题
- 高频需求优先于单点个人偏好
- 跑偏需求要有明确拦截机制

## 三、需求分级方法

建议按 A / B / C / D 四级管理。

### A：必须修

定义：

- 明显阻碍使用
- 影响核心链路
- 多人遇到且会直接导致放弃使用

典型例子：

- 发布卡住
- 活动报名失败
- 需求参与链路不通
- 页面理解严重混乱

### B：建议做

定义：

- 高频需求
- 不一定阻碍使用，但明显影响体验或转化
- 做了能明显提升内容、活动、需求三主线表现

典型例子：

- 标签筛选优化
- 我的参与页优化
- Banner 展示优化

### C：可观察

定义：

- 少量用户提出
- 目前影响有限
- 需要继续看频次和场景

典型例子：

- 某类排序方式
- 某个展示样式偏好
- 某个不影响主链路的小优化

### D：暂不做

定义：

- 与当前阶段目标不一致
- 资源投入大于短期收益
- 容易让项目跑偏

典型例子：

- 复杂聊天系统
- 电商商城
- 复杂会员体系

## 四、每条需求必须记录的信息

每条需求建议至少记录以下字段：

| 字段 | 说明 |
| --- | --- |
| 需求编号 | 例如 `ITR-001` |
| 需求标题 | 一句话概括 |
| 来源 | 谁提的、来自哪里 |
| 使用场景 | 用户在什么情况下提出 |
| 频次 | 提到次数或出现次数 |
| 影响范围 | 影响多少用户、哪个模块 |
| 当前分级 | A / B / C / D |
| 建议处理方式 | 修复 / 优化 / 观察 / 暂不做 |

## 六、判断标准建议

### 6.1 先判断是不是“阻碍使用”

如果答案是“会导致用户直接卡住或放弃”，优先考虑 A。

### 6.2 再判断是不是“高频”

如果多人重复提出，优先考虑 B。

### 6.3 再判断是不是“当前阶段该做”

如果需求虽好，但明显偏离内容、活动、需求三主线，优先考虑 D。

## 七、建议的需求池结构

### 7.1 按分级建池

建议维护 4 个池：

- A 池：必须修
- B 池：建议做
- C 池：观察池
- D 池：暂不做池

### 7.2 按来源标记

建议再打来源标签：

- 内测反馈
- 首轮冷启动数据
- 官方观察
- 合作方建议
- 用户直接反馈

### 7.3 按主线标记

建议按业务主线归类：

- 内容主线
- 活动主线
- 需求主线
- 我的页 / 用户资料
- 运营支持

## 八、建议处理方式定义

| 处理方式 | 说明 |
| --- | --- |
| 立即修复 | 进入近期开发或修复排期 |
| 进入候选 | 放入下一版本候选范围 |
| 持续观察 | 先继续收样本，不马上做 |
| 明确不做 | 放入跑偏需求或暂不做列表 |

## 九、推荐评审流程

### 第 1 步：统一收集

- 所有反馈先进入同一个需求池

### 第 2 步：去重合并

- 合并重复表达的相同需求

### 第 3 步：判断频次和影响

- 看是不是高频
- 看是不是影响核心链路

### 第 4 步：分级

- 分入 A / B / C / D

### 第 5 步：给建议处理方式

- 修复
- 候选
- 观察
- 暂不做

## 十、推荐示例

| 需求标题 | 来源 | 场景 | 频次 | 影响范围 | 分级 | 建议处理 |
| --- | --- | --- | --- | --- | --- | --- |
| 发布成功反馈不够明显 | 内测用户 | 发帖后不知道是否发成功 | 高 | 内容主线 | A | 立即优化 |
| 希望首页标签筛选更清楚 | 多位内容型用户 | 浏览首页内容时切换困难 | 中 | 内容主线 | B | 进入候选 |

| 希望上线私聊聊天系统 | 个别用户 | 浏览产品后提出 | 低 | 全局 | D | 暂不做 |

## 十一、常见误区提醒

### 不建议

- 谁声音大就先做谁的
- 单个用户提一次就立刻排开发
- 不区分“缺陷”和“需求”
- 不区分“高频需求”和“个体偏好”

### 建议

- 核心链路问题优先
- 高频需求优先
- 与阶段目标一致的需求优先

## 十二、上线前或每轮复盘后建议动作

- 更新需求池分级
- 把 A 池和 B 池同步到下一轮候选
- 把 D 池同步到跑偏需求清单
- 定期回看 C 池，避免遗漏逐渐变高频的需求
