# 粤次元君 - 冷启动阶段跑偏需求识别清单 V1

## 一、目标

冷启动阶段最危险的不是没人提需求，而是大家会提很多“听起来很酷”的需求，结果把项目拉散。

本清单的目标是：

- 明确当前阶段不建议进入的方向
- 给团队讨论时一个统一依据
- 帮助大家在冷启动阶段持续聚焦

## 二、判断跑偏需求的标准

如果一个需求同时满足以下任意两条，就要高度警惕：

- 不直接增强内容、活动、需求三主线
- 开发与维护成本明显偏高
- 会显著增加产品复杂度
- 会改变平台定位
- 会抢走有限资源，影响当前核心闭环

## 三、当前阶段明确不建议进入的方向

### 3.1 复杂聊天系统

为什么会跑偏：

- 会把产品从“内容 + 活动 + 需求协作平台”拉向“即时社交平台”
- 研发、风控、审核、举报、消息系统成本都会大幅增加
- 当前阶段更应该先把内容、报名、参与三主线做顺

当前建议：

- 不做私聊系统
- 不做复杂消息会话流

## 四、电商商城

### 4.1 为什么会跑偏

- 会把平台拉向交易平台
- 需要商品、订单、支付、售后、履约等完整能力
- 与当前冷启动最关键的内容、活动、需求三主线并不一致

当前建议：

- 不做商城
- 不做周边售卖闭环

## 五、复杂付费会员

### 5.1 为什么会跑偏

- 产品还在冷启动，核心价值尚未验证，过早做会员容易南辕北辙
- 需要权益设计、付费转化、支付、续费和长期运营策略
- 当前更重要的是让用户愿意来、愿意发、愿意参与

当前建议：

- 不做复杂会员体系
- 不做多层级付费权益设计

## 六、全城生活方式平台

### 6.1 为什么会跑偏

- 会从“二次元同好社区”被拉成“本地生活信息平台”
- 类目会迅速扩散，失去圈层聚焦
- 运营成本和内容范围都会失控

当前建议：

- 不扩到吃喝玩乐全城生活服务
- 不做泛本地活动聚合平台

## 七、泛摄影平台

### 7.1 为什么会跑偏

- 项目当前是围绕二次元内容、活动与协作，不是纯摄影社区
- 如果过度扩成泛摄影平台，会稀释 cosplay、妆造、活动招募等核心特色

当前建议：

- 摄影内容可以有，但要服务二次元场景
- 不做“所有摄影都能来”的泛摄影定位

## 八、泛社交平台

### 8.1 为什么会跑偏

- 一旦目标变成泛社交，功能需求会无限膨胀
- 会迅速拉向聊天、动态、关系链、推荐、群组等重系统
- 当前产品更适合围绕内容、活动、招募形成轻社区

当前建议：

- 不做泛陌生人社交
- 不做复杂关系链扩张

## 九、超重后台系统

### 9.1 为什么会跑偏

- 冷启动阶段最需要的是快速运营与快速迭代，不是先做一个大而全后台
- 重后台会消耗大量研发资源
- 当前后台应以最小运营和最小风控为主

当前建议：

- 只做最小必要后台能力
- 不做复杂权限系统、复杂工单流、复杂运营中台

## 十、其他需要警惕的方向

### 10.1 复杂推荐算法

风险：

- 冷启动内容量不够时，复杂推荐收益很有限
- 容易过早进入高复杂度建设

### 10.2 大规模多城市扩张设计

风险：

- 当前更适合先聚焦一个核心区域或明确圈层
- 过早多城市化会让内容和活动供给难以支撑

### 10.3 复杂搜索系统

风险：

- 冷启动阶段内容池有限
- 复杂搜索容易做重，但不一定立刻带来价值

## 十一、什么需求不算跑偏

以下需求通常不算跑偏，反而是合理增强项：

- 收藏
- 关注
- 标签筛选优化
- 我的参与页优化
- Banner 展示优化
- 官方活动展示增强
- 举报入口
- 轻量搜索入口

这些需求仍然围绕：

- 内容互动增强
- 活动体验增强
- 需求协作增强

## 十二、讨论时建议使用的判断问题

当团队讨论一个新需求时，先问：

1. 这个需求是否直接增强内容、活动或需求主线？
2. 它是否能帮助冷启动阶段留人、发内容、报名、参与？
3. 它是否会显著增加系统复杂度？
4. 如果现在做它，会不会挤占更重要的事情？

如果答案偏向“不能增强主线、复杂度高、会挤占资源”，大概率就是跑偏需求。

## 十三、建议处理方式

对于跑偏需求，不建议直接争论对错，建议统一处理为：

- 记录进观察或暂不做清单
- 标记“不属于当前阶段”
- 等核心主线跑顺、冷启动验证完成后再重新评估

## 十四、上线前检查 Checklist

- [ ] 团队知道当前阶段主线是什么
- [ ] 团队知道哪些方向明确不建议做
- [ ] 新需求讨论时会先判断是否跑偏
- [ ] 跑偏需求不会直接进入开发排期
- [ ] 资源优先留给内容、活动、需求三主线

## 十五、可直接丢给 Trae 的 Prompt

请为“粤次元君”整理冷启动阶段的“跑偏需求识别清单”。

要求：

1. 列出当前阶段明确不建议进入的方向
2. 说明为什么这些需求会让项目跑偏
3. 输出 `out-of-scope-watchlist-v1.md`
4. 目标是帮助团队在冷启动阶段保持聚焦
