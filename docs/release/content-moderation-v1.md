# 粤次元君 - 最小内容审核与风控准备方案 V1

## 一、目标

“粤次元君”当前是 UGC 产品，MVP 阶段不做复杂审核系统，但必须具备最小可控能力，确保上线后不是完全裸奔。

本方案只定义 MVP 必须具备的最小治理能力，不引入复杂机审、工单流或多级审批。

## 二、当前现状判断

基于当前仓库检查结果：

- 帖子、需求、活动已有基础发布与展示链路
- 后端已存在部分业务状态字段：
  - 活动：`UPCOMING / ONGOING / ENDED`
  - 需求：`OPEN / CLOSED`
- 帖子实体当前未看到独立内容状态字段
- 前端暂未发现“举报”入口
- 文档中已明确后台最小能力包含“强制下架 / 删除”
- 当前更接近“内容可发可看”，但“违规后如何处理”尚未在产品层闭环

## 三、MVP 最小风控原则

- 不做复杂审核流，但要有“可拦截、可隐藏、可下架、可追踪”的基础能力
- 优先处理高风险内容：涉黄、涉政、违法违规、引流诈骗、人身攻击、未成年人风险
- 默认以人工处理为主，先保证有操作入口，再逐步自动化
- 用户侧要有明确提示，避免“无反馈式失败”

## 四、内容状态建议

### 4.1 建议统一的最小状态模型

建议帖子、需求、活动都逐步统一到以下最小内容状态：

| 状态 | 含义 | 用户可见性 | 适用说明 |
| --- | --- | --- | --- |
| `PUBLISHED` | 正常发布 | 可见 | 默认公开状态 |
| `PENDING` | 待审核 / 待人工确认 | 仅本人可见或前台不可见 | 用于高风险内容、首批风控兜底 |
| `HIDDEN` | 已隐藏 / 已下架 | 不可见 | 举报成立、后台手动下架、敏感内容拦截后使用 |

### 4.2 当前项目落地建议

| 内容类型 | 当前状态 | 建议补充 |
| --- | --- | --- |
| 帖子 | 无独立内容状态字段 | 优先增加 `status`，至少支持 `PUBLISHED / HIDDEN`，有余力再补 `PENDING` |
| 需求 | 当前为 `OPEN / CLOSED` | 在业务状态之外再增加内容可见状态，避免“招募关闭”和“违规下架”混用 |
| 活动 | 当前为 `UPCOMING / ONGOING / ENDED` | 同样建议补充内容可见状态，避免用活动进度状态承担审核语义 |

建议结论：

- 业务状态和审核状态不要混在一个字段里
- MVP 若来不及全量改造，至少先给帖子与需求补 `HIDDEN`
- 后台下架动作应直接落到“内容可见状态”，而不是删除数据

## 五、举报入口预留建议

### 5.1 MVP 最小入口位置

建议优先预留以下入口：

- 帖子详情页右上角：举报
- 需求详情页右上角：举报
- 活动详情页右上角：举报 / 反馈

如果当前版本人力有限，最小可接受方案：

- 先做一个统一“举报”按钮
- 点击后弹出举报原因选择 + 补充说明输入框
- 提交后只要能写入数据库或日志表即可

### 5.2 举报原因建议

建议首版提供固定选项：

- 色情低俗
- 广告引流
- 诈骗 / 交易风险
- 人身攻击
- 违法违规
- 其他

### 5.3 MVP 处理方式

- 用户提交举报后返回“已收到，我们会尽快处理”
- 后端存储举报记录
- 管理后台暂时不做复杂工单列表也可接受
- 运营可先通过数据库 / 简易后台查看，再手动处理

## 六、敏感内容处理思路

### 6.1 优先拦截范围

MVP 阶段建议优先关注以下内容：

- 色情低俗图片或文案
- 明显违法违规内容
- 导流到外部社群、私人联系方式、交易诈骗
- 大量重复刷屏、灌水、营销文案
- 对个人的辱骂、攻击、骚扰

### 6.2 最小处理策略

| 风险级别 | 处理方式 |
| --- | --- |
| 高风险 | 直接拦截发布或发布后立即设为 `HIDDEN` |
| 中风险 | 先进入 `PENDING`，由人工确认 |
| 低风险 | 允许发布，但保留举报与后台下架能力 |

### 6.3 MVP 可执行建议

- 文本先做基础关键词拦截，宁可少量误伤，也不要完全无防线
- 图片阶段不强制接复杂审核服务，但建议预留图片审核接入位
- 对重复发布、明显广告号可先通过人工处理
- 被隐藏内容保留原始记录，方便追查与申诉

## 七、下架 / 隐藏能力建议

### 7.1 后台最小能力

根据现有产品文档，后台最小能力已具备方向，建议确认以下动作：

- 帖子下架
- 需求下架
- 活动隐藏
- 用户账号临时禁用发布

### 7.2 MVP 最小动作定义

| 动作 | 建议行为 |
| --- | --- |
| 下架内容 | 将审核状态改为 `HIDDEN`，前台不再展示 |
| 隐藏内容 | 与下架同义，统一走 `HIDDEN` |
| 关闭需求 | 仍表示业务结束，不等同违规处理 |
| 删除内容 | MVP 不建议物理删除，优先软隐藏 |

### 7.3 操作留痕建议

即使不做完整审计日志，也建议至少记录：

- 操作人
- 操作时间
- 操作对象
- 原因

这样能避免后续出现“内容为什么消失了”无法追溯。

## 八、用户提示文案建议

### 8.1 发布被拦截

- 你的内容包含可能不适宜展示的信息，请修改后再试
- 检测到疑似广告、引流或违规内容，当前无法发布
- 为保障社区环境，当前内容未通过校验，请调整后重新提交

### 8.2 内容进入待处理

- 你的内容已提交，正在处理中，请稍后查看
- 当前内容需要进一步确认，审核完成后会自动展示

### 8.3 内容被下架

- 你的内容因违反社区规则已被隐藏，如有疑问可联系客服反馈
- 该内容暂不可见，原因可能为被举报或平台处理

### 8.4 举报提交成功

- 举报已提交，感谢你的反馈
- 我们已收到举报信息，会尽快核查处理

## 九、推荐的 MVP 处理流程

### 9.1 发布流程

1. 用户提交内容
2. 先做基础校验与敏感词检查
3. 低风险内容直接 `PUBLISHED`
4. 高风险内容直接拦截或记为 `HIDDEN`
5. 灰区内容进入 `PENDING`

### 9.2 举报流程

1. 用户在详情页点击举报
2. 选择原因并提交
3. 后端记录举报信息
4. 运营人工查看
5. 必要时将内容改为 `HIDDEN`

### 9.3 下架流程

1. 运营或管理员发现违规内容
2. 执行下架
3. 内容前台隐藏
4. 用户看到相应提示
5. 后台保留记录

## 十、上线前最小 Checklist

- [ ] 帖子有可用的审核状态字段或可隐藏能力
- [ ] 需求有“业务关闭”和“违规隐藏”的区分方案
- [ ] 举报入口至少在详情页预留
- [ ] 举报原因有固定选项
- [ ] 后端可接收举报信息
- [ ] 管理端或运维侧具备最小手动下架能力
- [ ] 发布失败、拦截、下架都有用户提示文案
- [ ] 高风险内容至少可人工处理，不会长期裸露在线上

## 十一、建议优先级

### P0：上线前必须具备

- 举报入口预留
- 后台手动下架能力
- 帖子 / 需求最少支持 `HIDDEN`
- 拦截与下架提示文案

### P1：建议尽快补齐

- `PENDING` 状态
- 敏感词基础拦截
- 举报记录查询能力

### P2：后续演进

- 图片审核服务接入
- 用户处罚等级
- 风险账号识别
- 审核工单流
