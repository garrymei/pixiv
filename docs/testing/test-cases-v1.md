# 粤次元君 - 阶段 5 测试用例清单 V1

本文档用于沉淀“粤次元君”阶段 5 的 MVP 核心测试用例。测试用例按主线分组，可直接用于手工测试执行、缺陷记录和回归验证。

---

## 一、优先级说明

- **P0**：核心链路，失败会直接影响 MVP 可用性。
- **P1**：核心体验相关，失败不一定阻断链路，但会显著影响使用。
- **P2**：细节或边界问题，可记录并安排后续修复。

---

## 二、登录与用户状态

### TC-AUTH-001 自动建立登录态

- **测试目标**：验证应用启动后可建立有效会话，并在需要鉴权的接口上正常使用。
- **前置条件**：应用处于 real 模式；后端登录接口和鉴权接口可用；本地无异常缓存。
- **操作步骤**：
  1. 清理本地 token 与用户缓存。
  2. 进入应用首页。
  3. 等待首页和用户态相关接口初始化完成。
  4. 进入“我的”页面。
- **预期结果**：
  1. 应用可自动完成登录或会话初始化。
  2. 首页可正常加载，不出现未授权导致的白屏。
  3. “我的”页面能展示真实用户昵称、头像或默认占位。
  4. 本地已保存有效 token 与会话用户信息。
- **优先级**：P0

### TC-AUTH-002 鉴权失效后的恢复能力

- **测试目标**：验证 token 失效时，请求层具备恢复能力或至少给出明确失败提示。
- **前置条件**：应用已登录；可手动构造失效 token。
- **操作步骤**：
  1. 将本地 token 替换为无效值。
  2. 进入需要鉴权的数据页面，如“我的”页面。
  3. 触发一次需要鉴权的提交操作，如编辑资料保存。
- **预期结果**：
  1. 请求层不会因 token 失效直接导致页面卡死。
  2. 系统会尝试恢复会话或返回清晰错误提示。
  3. 页面保留可继续操作的状态，不出现不可恢复异常。
- **优先级**：P1

## 三、内容主线

### TC-CONTENT-001 首页浏览正常加载

- **测试目标**：验证首页内容流在正常场景下可加载并展示真实数据。
- **前置条件**：后端存在帖子与需求数据；应用处于 real 模式。
- **操作步骤**：
  1. 进入首页。
  2. 观察首次加载态。
  3. 检查帖子流展示信息。
  4. 切换到需求 Tab。
- **预期结果**：
  1. 首次进入显示 loading 状态。
  2. 帖子卡片展示封面、标题、作者、点赞数、评论数、标签等信息。
  3. 需求列表展示类型、标题、预算、地点、作者、状态等信息。
  4. 无数据时出现 empty 状态，请求失败时出现 error 状态和重试入口。
- **优先级**：P0

### TC-CONTENT-002 帖子详情字段展示一致

- **测试目标**：验证从首页进入帖子详情后，字段完整且与列表数据保持基本一致。
- **前置条件**：首页存在至少一条帖子数据。
- **操作步骤**：
  1. 在首页记录一条帖子卡片的标题、作者、点赞数、评论数。
  2. 点击进入帖子详情。
  3. 检查详情页标题、作者、时间、正文、图片、标签、点赞数、评论区。
- **预期结果**：
  1. 可正常进入详情页，无白屏。
  2. 标题、作者等基础字段与列表一致。
  3. 详情页可展示正文、图片数组、标签、评论列表。
  4. 页面具备 loading、error、empty 处理能力。
- **优先级**：P0

### TC-CONTENT-003 发布帖子成功闭环

- **测试目标**：验证发布帖子主链路可从图片上传一路完成到首页可见。
- **前置条件**：用户已登录；准备至少 1 张可上传图片。
- **操作步骤**：
  1. 进入发布帖子页面。
  2. 选择图片并等待上传成功。
  3. 输入标题、内容，选择标签，可选填写定位。
  4. 点击发布。
  5. 发布成功后返回首页。
  6. 在首页或我的发布中查找新帖子。
- **预期结果**：
  1. 图片上传期间显示上传中状态，成功后给出成功反馈。
  2. 表单通过校验后可提交。
  3. 发布成功提示明确。
  4. 新帖子可在首页或我的发布中看到。
  5. 新帖子详情可正常打开，字段完整。
- **优先级**：P0

### TC-CONTENT-004 发布帖子表单校验

- **测试目标**：验证发布帖子缺少必要信息时会阻止提交。
- **前置条件**：进入发布帖子页面。
- **操作步骤**：
  1. 不填标题直接提交。
  2. 不填内容直接提交。
  3. 不选标签直接提交。
  4. 存在上传失败图片时尝试提交。
- **预期结果**：
  1. 页面对缺失项给出明确校验提示。
  2. 提交被阻止，不会产生脏数据。
  3. 上传失败图片未处理前不可发布。
- **优先级**：P1

### TC-CONTENT-005 帖子点赞状态变更

- **测试目标**：验证点赞、取消点赞、计数变化与页面回显逻辑正确。
- **前置条件**：存在可点赞帖子；用户已登录。
- **操作步骤**：
  1. 进入帖子详情。
  2. 记录当前点赞状态和点赞数。
  3. 点击点赞按钮。
  4. 再次点击取消点赞。
  5. 返回首页查看对应卡片。
- **预期结果**：
  1. 点赞后按钮状态更新，点赞数加 1。
  2. 取消点赞后按钮状态恢复，点赞数减 1。
  3. 详情页和首页卡片的点赞结果保持一致。
  4. 操作失败时有明确提示并回滚 UI 状态。
- **优先级**：P0

### TC-CONTENT-006 评论发布与评论数同步

- **测试目标**：验证评论成功后，评论列表与评论数同步更新。
- **前置条件**：存在可评论帖子；用户已登录。
- **操作步骤**：
  1. 进入帖子详情。
  2. 记录当前评论数。
  3. 输入评论内容并提交。
  4. 观察评论列表顶部是否新增评论。
  5. 返回首页后再次进入详情检查评论数。
- **预期结果**：
  1. 空评论不可提交。
  2. 评论成功后有成功提示。
  3. 新评论立即显示在评论列表中。
  4. 评论数随之增加，并与首页卡片保持一致。
- **优先级**：P0

## 四、活动主线

### TC-EVENT-001 活动列表展示正确

- **测试目标**：验证活动列表能够展示真实活动数据和正确状态。
- **前置条件**：后端存在活动数据，包含至少一条官方活动或资讯活动。
- **操作步骤**：
  1. 进入活动列表页。
  2. 观察 loading、empty、error 表现。
  3. 检查卡片中的标题、时间、地点、状态、费用、名额、截止时间。
  4. 点击任意活动进入详情。
- **预期结果**：
  1. 列表页可正常展示活动卡片。
  2. 活动状态显示正确。
  3. 点击卡片可进入对应详情页。
  4. 空数据与失败场景有对应页面反馈。
- **优先级**：P0

### TC-EVENT-002 活动详情展示完整

- **测试目标**：验证活动详情页字段完整、状态可理解。
- **前置条件**：存在活动详情数据。
- **操作步骤**：
  1. 进入活动详情页。
  2. 检查标题、类型、报名状态、时间、地点、主办方、名额、截止时间、费用、描述。
  3. 观察底部按钮文案和可点击状态。
- **预期结果**：
  1. 所有核心字段可正常展示。
  2. 活动类型与状态文案正确。
  3. 不可报名场景会明确给出原因。
- **优先级**：P0

### TC-EVENT-003 官方活动报名成功

- **测试目标**：验证官方活动可成功报名，并在我的活动中回显。
- **前置条件**：存在未报名、未截止、未结束的官方活动；用户已登录。
- **操作步骤**：
  1. 进入官方活动详情页。
  2. 点击“立即报名”。
  3. 观察按钮状态和成功提示。
  4. 进入“我的活动”页面查看结果。
- **预期结果**：
  1. 报名请求成功。
  2. 详情页显示已报名状态。
  3. “我的活动”中出现对应活动记录。
  4. 重复进入详情时仍保持已报名状态。
- **优先级**：P0

### TC-EVENT-004 非可报名活动的拦截逻辑

- **测试目标**：验证资讯活动、已截止活动、已结束活动不可报名。
- **前置条件**：存在资讯型活动或已截止/已结束活动。
- **操作步骤**：
  1. 进入资讯型活动详情。
  2. 观察按钮状态和提示文案。
  3. 分别进入已截止和已结束活动详情。
  4. 尝试点击报名按钮。
- **预期结果**：
  1. 资讯型活动不可报名，提示“暂不支持报名”。
  2. 已截止活动不可报名，提示“报名已截止”。
  3. 已结束活动不可报名，提示“活动已结束”。
  4. 不会发出错误的成功请求。
- **优先级**：P1

### TC-EVENT-005 我的活动列表回显

- **测试目标**：验证已报名活动可在“我的活动”中稳定查看。
- **前置条件**：当前用户至少有 1 条报名成功记录。
- **操作步骤**：
  1. 进入“我的活动”页面。
  2. 检查活动列表展示内容。
  3. 点击进入任一详情页。
- **预期结果**：
  1. 我的活动列表可正常加载。
  2. 活动时间、标题、地点等基础信息正确。
  3. 点击后可进入对应详情页。
  4. 无数据或失败时显示空态/错误态。
- **优先级**：P0

## 五、需求主线

### TC-DEMAND-001 需求列表展示正确

- **测试目标**：验证需求列表可正确展示真实需求数据。
- **前置条件**：后端存在需求数据。
- **操作步骤**：
  1. 在首页需求 Tab 或需求相关页面查看需求列表。
  2. 检查类型、标题、预算、地点、时间、作者、状态字段。
  3. 点击任一需求进入详情页。
- **预期结果**：
  1. 需求卡片字段完整且可理解。
  2. 状态展示与详情一致。
  3. 可正常进入需求详情页。
- **优先级**：P0

### TC-DEMAND-002 需求详情展示完整

- **测试目标**：验证需求详情页能够完整展示需求信息与参与状态。
- **前置条件**：存在需求详情数据。
- **操作步骤**：
  1. 进入需求详情页。
  2. 检查标题、类型、预算、地点、时间、状态、人数、截止时间、作者、描述。
  3. 观察参与按钮状态与文案。
- **预期结果**：
  1. 详情字段完整展示。
  2. 状态、截止时间、人数信息与业务实际一致。
  3. 按钮文案会根据已申请、已关闭、已截止等状态变化。
- **优先级**：P0

### TC-DEMAND-003 发布需求成功闭环

- **测试目标**：验证发布需求成功后，可在我的需求中查看到新记录。
- **前置条件**：用户已登录。
- **操作步骤**：
  1. 进入发布需求页面。
  2. 填写类型、标题、描述、时间、截止时间、地点、预算、人数、联系说明。
  3. 点击提交。
  4. 进入“我的需求”页面查看结果。
- **预期结果**：
  1. 表单校验通过后可成功提交。
  2. 发布成功后出现成功提示。
  3. “我的需求”中出现新发布需求。
  4. 进入详情后字段与提交内容基本一致。
- **优先级**：P0

### TC-DEMAND-004 发布需求表单校验

- **测试目标**：验证发布需求必填项校验完善。
- **前置条件**：进入发布需求页面。
- **操作步骤**：
  1. 不选择类型直接提交。
  2. 不填标题、描述、时间、截止时间、地点分别提交。
  3. 不选择预算和人数提交。
- **预期结果**：
  1. 页面对每个缺失项给出明确提示。
  2. 不满足条件时不会提交成功。
  3. 用户可根据提示快速修正输入。
- **优先级**：P1

### TC-DEMAND-005 参与需求成功回显

- **测试目标**：验证用户可参与开放中的需求，并在相关我的页面看到结果。
- **前置条件**：存在开放中、未截止的需求；用户已登录；当前用户未参与过该需求。
- **操作步骤**：
  1. 进入需求详情页。
  2. 点击“感兴趣”或参与按钮。
  3. 观察提交反馈。
  4. 进入“我的参与”或相关我的页面查看数据。
- **预期结果**：
  1. 参与成功后出现成功提示。
  2. 当前详情页状态更新为已申请。
  3. 我的相关页面中出现对应需求记录。
  4. 重复进入详情仍保持已申请状态。
- **优先级**：P0

### TC-DEMAND-006 不可参与需求的拦截逻辑

- **测试目标**：验证已关闭、已截止、已参与的需求不会重复参与。
- **前置条件**：存在对应状态的需求记录。
- **操作步骤**：
  1. 分别进入已关闭、已截止、已参与的需求详情。
  2. 观察按钮文案和可点击状态。
  3. 尝试触发参与行为。
- **预期结果**：
  1. 已关闭需求不可参与。
  2. 已截止需求不可参与。
  3. 已参与需求不可重复提交。
  4. 页面给出明确提示，不产生错误成功反馈。
- **优先级**：P1

### TC-DEMAND-007 我的需求列表回显

- **测试目标**：验证当前用户发布过的需求可在“我的需求”中查看。
- **前置条件**：当前用户已有至少 1 条需求发布记录。
- **操作步骤**：
  1. 进入“我的需求”页面。
  2. 检查列表字段展示。
  3. 点击进入详情。
- **预期结果**：
  1. 我的需求可正常加载。
  2. 列表字段与详情字段基本一致。
  3. 空数据与错误场景有对应反馈。
- **优先级**：P0

## 六、我的主线

  2. 检查封面、标题、作者、统计数据。

  1. 列表可正常加载。

  3. 详情页内容与我的发布卡片基本一致。
  4. 空态和错误态有清晰反馈。
- **优先级**：P0
