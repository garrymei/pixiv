# 粤次元君 - 阶段 5 测试范围定义 V1

本文档用于定义“粤次元君”在阶段 5 的 MVP 测试范围，确保测试工作围绕核心业务闭环展开，不遗漏主链路，不扩散到当前版本边界外。

---

## 一、测试目标

阶段 5 测试聚焦以下目标：

1. 验证 MVP 主链路是否真实可用，可从用户入口一路完成关键业务动作。
2. 验证前后端联调后的字段、状态、时间、图片、鉴权行为是否稳定。
3. 验证页面在正常、异常、空数据、提交反馈等场景下的用户体验是否可接受。
4. 为阶段 5 后续测试执行、缺陷记录、上线准备提供统一测试边界。

---

## 二、测试范围原则

### 2.1 本次纳入范围

- 只覆盖 MVP 核心功能与主链路。
- 只覆盖当前真实接入后的前端页面、service 层、接口联调结果。
- 重点覆盖“浏览、提交、状态变化、回显、我的数据可见”这类闭环行为。
- 同时覆盖成功路径、失败路径、空态路径、鉴权路径。

### 2.2 本次暂不纳入范围

- 管理后台与运营后台能力。
- 推荐算法、搜索、多级评论、私信、消息通知等二期能力。
- 性能压测、安全攻防、灰度发布、监控告警等专项测试。
- 多端差异专项兼容测试。

---

## 三、测试对象

本轮测试对象包括：

- 前端核心页面与页面交互状态。
- 前端统一请求层、token 管理、mock/real 切换能力。
- 后端已被真实消费的接口与接口返回字段。
- 上传接口与依赖上传结果的发布行为。

---

## 四、必测业务主线

### 4.1 登录与用户状态

### 覆盖内容

- 自动登录或初始化会话是否成功。
- token 是否正确保存、透传、失效后是否可恢复。
- 已登录状态下，依赖鉴权的页面和操作是否可正常执行。
- 用户信息是否能在“我的页”和相关消费页面正确展示。

### 重点验证

- 首次进入应用可建立有效用户会话。
- 鉴权接口在真实模式下可被正常访问。
- 登录态失效时有合理兜底，不出现页面卡死或无提示失败。

---

### 4.2 内容主线

### 覆盖功能

- 首页浏览
- 帖子详情
- 发布帖子
- 点赞
- 评论

### 重点行为

- 首页能正确加载帖子/需求卡片，不同状态下展示 loading、empty、error。
- 点击帖子可进入详情页，详情字段与列表字段保持一致。
- 发布帖子时图片上传、表单校验、发布成功反馈、列表可见形成闭环。
- 点赞状态、点赞数能正确变化并可回显。
- 评论提交成功后评论列表即时更新，评论数同步变化。

### 重点风险

- 图片上传成功但发布失败，或发布成功但列表未刷新。
- 点赞/评论后详情页与列表页数据不一致。
- 时间、图片、作者信息、统计字段展示异常。

---

### 4.3 活动主线

### 覆盖功能

- 活动列表
- 活动详情
- 活动报名
- 我的活动

### 重点行为

- 活动列表可区分活动资讯与官方活动。
- 活动详情可正确展示时间、地点、主办方、名额、截止时间、状态。
- 仅可对可报名活动执行报名操作，报名后状态更新并能在“我的活动”中看到。
- 已报名、已截止、已结束、不可报名等状态有明确反馈。

### 重点风险

- 报名成功但“我的活动”无数据。
- 资讯型活动误出现可报名入口。
- 报名状态接口与详情按钮状态不一致。

---

### 4.4 需求主线

### 覆盖功能

- 需求列表
- 需求详情
- 发布需求
- 参与需求
- 我的需求

### 重点行为

- 需求列表能正确展示类型、预算、时间、地点、作者、状态。
- 需求详情信息完整，参与按钮与当前状态一致。
- 发布需求表单校验、提交成功、我的需求回显形成闭环。
- 参与需求成功后状态可回显，并能在“我的参与”或相关我的页面看到。

### 重点风险

- 需求状态、截止时间、人数限制判断错误。
- 参与成功但列表不刷新或我的相关页不回显。
- 发布需求后的详情字段与列表字段不一致。

---

### 4.5 我的主线

### 覆盖功能

- 我的页面
- 编辑资料
- 我的发布

### 重点行为

- 我的页面聚合数据真实化，展示用户信息和统计入口。
- 编辑资料后能保存成功并回显到我的页面及相关作者信息展示处。
- 我的发布能展示当前用户真实发布内容，并支持进入详情检查数据一致性。

### 重点风险

- 更新资料成功但页面缓存未刷新。
- 我的页统计信息与实际列表不一致。
- 我的发布为空态、错误态缺失。

---

## 五、通用验证维度

所有纳入范围的功能，除功能本身外，还需要从以下维度验证：

| 维度 | 验证点 |
| --- | --- |
| 页面状态 | 是否具备 loading / empty / error / success 反馈 |
| 数据一致性 | 列表与详情字段是否一致，提交后是否能回显 |
| 鉴权稳定性 | requireAuth 接口是否自动带 token，失效时是否可恢复 |
| 表单校验 | 必填项、非法输入、重复提交是否处理正确 |
| 提交反馈 | 成功/失败是否有清晰提示，按钮是否有 loading 防重入 |
| 时间字段 | 毫秒时间戳与兼容字符串时间是否都能稳定展示 |
| 图片字段 | 封面图、图片数组、上传后返回地址是否正确可访问 |
| 状态字段 | 点赞、报名、参与、开放/关闭、进行中/结束等状态是否一致 |

---

## 六、优先级定义

| 优先级 | 定义 | 处理要求 |
| --- | --- | --- |
| P0 | 核心链路中断，直接影响 MVP 可用性 | 必须测试、必须修复 |
| P1 | 核心功能可用但存在明显异常或错误体验 | 必须测试，原则上阶段内修复 |
| P2 | 非主链路细节问题，不阻断核心业务 | 可在阶段内记录并排期处理 |

---

## 七、范围与页面映射

| 主线 | 页面/模块 | 核心动作 | 优先级 |
| --- | --- | --- | --- |
| 登录与用户状态 | 请求层 / 会话初始化 / 我的页 | 登录、鉴权、用户态回显 | P0 |
| 内容主线 | 首页、帖子详情、发布帖子 | 浏览、上传、发布、点赞、评论 | P0 |
| 活动主线 | 活动列表、活动详情、我的活动 | 浏览、报名、结果回显 | P0 |
| 需求主线 | 需求列表、需求详情、发布需求、我的需求 | 浏览、发布、参与、回显 | P0 |
| 我的主线 | 我的页、编辑资料、我的发布 | 信息聚合、编辑、回显 | P0 |

---

## 九、通过标准

满足以下条件即可视为阶段 5 测试范围定义完成：

1. 已覆盖登录、内容、活动、需求、我的五大主线。
2. 每条主线已明确页面范围、核心行为、重点风险。
3. 已定义统一优先级与通用验证维度。
4. 后续测试执行可以直接基于测试清单开展，不需要重新拆解范围。
