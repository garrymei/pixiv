# 粤次元君 (Yue Ci Yuan Jun) - 角色与权限设计 V1

本文档定义了 MVP 阶段所需的最小角色集合及其对应的权限边界，旨在避免后端接口越权漏洞，同时保持系统的轻量化，不引入过度复杂的 RBAC（基于角色的访问控制）模型。

---

## 1. 角色定义

MVP 阶段系统仅包含以下 4 种基础角色：

| 角色标识 | 角色名称 | 获取方式 | 说明 |
|---|---|---|---|
| `guest` | 游客 | 未登录/未授权微信 | 仅能浏览公开信息，无法互动。 |
| `user` | 普通用户 | 微信授权登录成功 | 社区的核心参与者，能发布和互动。 |
| `official` | 官方账号 | 后台手动分配标记 | 用于发布官方活动和重要通知的特殊用户。 |
| `admin` | 管理员 | 后台管理系统账号 | 拥有最高权限，管理全局内容和秩序。 |

> **实现建议**：在 `users` 表中，目前使用 `status` (禁用状态) 来控制。为了区分官方账号和管理员，建议在后续开发时（或在 Token Payload 中）隐式包含一个 `role` 字段（默认 `user`）。MVP 阶段，`admin` 可独立于 C 端用户表，存在于一个简单的后台管理账号体系中。

## 2. 权限矩阵 (Permissions Matrix)

以下矩阵明确了在 MVP 核心模块中，不同角色的可执行操作。

| 模块 / 操作 | `guest` (游客) | `user` (普通用户) | `official` (官方) | `admin` (管理员) |
|---|:---:|:---:|:---:|:---:|
| **浏览内容** (帖子/需求/活动) | ✅ | ✅ | ✅ | ✅ |
| **搜索/筛选列表** | ✅ | ✅ | ✅ | ✅ |
| **点赞 / 评论** | ❌ (拦截去登录)| ✅ | ✅ | ✅ |
| **发布作品 / 日常 (帖子)** | ❌ | ✅ | ✅ | ✅ |
| **发布需求** | ❌ | ✅ | ✅ | ✅ |
| **参与需求 (申请)** | ❌ | ✅ | ✅ | ❌ (不参与业务)|
| **报名官方活动** | ❌ | ✅ | ✅ | ❌ |
| **发布活动 (活动列表)** | ❌ | ❌ (无权限) | ✅ | ✅ |
| **编辑资料 (本人)** | ❌ | ✅ | ✅ | - |
| **删除内容 (帖子/评论)** | ❌ | ✅ (仅限自己的) | ✅ (仅限自己的) | ✅ (可删所有人) |
| **审核 / 下架他人内容** | ❌ | ❌ | ❌ | ✅ |
| **封禁/解封用户** | ❌ | ❌ | ❌ | ✅ |
| **管理 Banner** | ❌ | ❌ | ❌ | ✅ |

## 3. 权限控制落地策略

### 3.1 前端 (C 端小程序) 拦截
1. **浏览类拦截**：游客可正常浏览。
2. **交互类拦截**：
   - 任何触发“写操作”（发布、点赞、评论、报名）的按钮，点击时检查全局状态是否有 `Token`。
   - 若无，拦截并弹出“请先登录”的提示或直接跳转登录页。
3. **UI 隐藏**：
   - 帖子/评论的“删除”按钮，仅当 `当前登录用户 ID == 内容的 user_id` 时才渲染显示。

### 3.2 后端 (API) 拦截 (双重保险)
前端的隐藏和拦截是不安全的，后端必须做严格校验：
1. **Token 校验**：所有非公开 `GET` 接口（如 `/posts` `POST`），必须在中间件/拦截器中校验 Token 的合法性。
2. **数据越权校验 (核心)**：
   - 用户调用“删除帖子” `/posts/:id` `DELETE` 时，后端在执行 `UPDATE is_deleted=1` 前，**必须**查询该帖子，并判断 `post.user_id == jwt.user_id`。如果不等，直接返回 `403 Forbidden`。
3. **角色越权校验**：
   - 只有携带 `admin` 或 `official` 标识的 Token 才能调用 `/events` 的 `POST`（发布活动）接口。普通用户调用直接返回 `403`。
