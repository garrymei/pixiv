# 粤次元君 (Yue Ci Yuan Jun) - 后台管理范围定义 V1

本文档旨在明确 MVP 阶段（第一版）后台管理系统的**最小必要范围**。我们不追求在第一期就做一个大而全的 B 端系统，而是仅提供支撑 C 端小程序正常运转的最基础能力。

> **核心原则**：能直接改数据库解决的低频操作，就不做后台页面；必须高频操作的，才做后台页面。

---

## 1. Banner 管理 (运营位)
- **目标**: 运营人员能够随时更换小程序首页的置顶轮播图，而不依赖技术发版。
- **最小页面**: Banner 列表页（含新增/编辑弹窗）。
- **最小操作**:
  - 上传 Banner 图片。
  - 填写跳转链接（H5 URL 或 小程序内部路径）。
  - 设置排序权重。
  - 上下架控制（启用/禁用）。
- **依赖数据表**: `banners`（第一版数据库设计中预留或直接复用 `events` 表的置顶字段，建议独立 `banners` 表）。

## 2. 活动发布与状态管理 (官方活动)
- **目标**: 官方可以通过后台发布漫展资讯或官方活动，并管理报名进度。
- **最小页面**: 
  - 活动列表页。
  - 活动发布/编辑页（支持富文本/Markdown，设置时间、地点、人数限制）。
  - 报名人员查看页。
- **最小操作**:
  - 创建新活动（存入 `events` 表）。
  - 手动修改活动状态（例如提前结束报名）。
  - 导出报名人员列表（生成基础的 CSV/Excel，或仅列表展示）。
- **依赖数据表**: `events`, `event_registrations`。

## 3. 内容审核 (最基础版)
- **目标**: 应对合规风险，能快速下架违规的帖子或需求。
- **最小页面**: 
  - 内容管理列表（包含帖子列表和需求列表，支持按时间倒序或被举报次数倒序）。
- **最小操作**:
  - 列表展示帖子图文内容。
  - **强制下架/删除** 按钮（点击后将 `is_deleted` 置为 1 或修改 `status`）。
  - （可选）封禁发布该内容的用户。
- **依赖数据表**: `posts`, `demands`, `users`。
- **MVP不做的**: 不做机审接入，不做复杂的工单流转，发现违规直接人工下架。

## 4. 用户基础管理
- **目标**: 处理恶意用户，或将普通用户提升为“官方账号”。
- **最小页面**: 用户列表页。
- **最小操作**:
  - 按昵称或 ID 搜索用户。
  - **封禁/解封**用户（修改 `status` 字段）。
  - **标记身份**（给特定用户打上 `official` 标签，使其能发官方活动）。
- **依赖数据表**: `users`。

## 5. 举报查看 (预留)
- **目标**: 收集 C 端用户的反馈，发现违规内容。
- **MVP 策略**: 第一版可仅提供后端 API 接收 C 端举报请求并存入数据库，后台暂时不做专门的工单处理页面。管理员定期直接查看数据库，或者做一个只读的简单列表。

---

## 总结
**MVP 后台仅需要 4 个核心页面（菜单）：**
1. 首页 Banner 设置
2. 官方活动管理
3. 违规内容下架（帖子/需求）
4. 用户封禁与角色分配
