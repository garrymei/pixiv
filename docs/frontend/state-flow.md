# 粤次元君 (Yue Ci Yuan Jun) - 前端状态流与数据流设计

本文档规范了前端（基于微信小程序/Taro）的数据存储、状态流转和缓存策略，旨在避免后续开发中出现状态不同步、数据乱传的问题，降低联调复杂度。

---

## 1. 状态分类规则

前端状态严格划分为两类：**全局状态** 和 **页面本地状态**。

### 1.1 全局状态 (Global State)
> 存放在 Redux / Pinia / Zustand (视最终技术栈而定) 或小程序 `getApp().globalData` 中，并按需持久化到 `wx.setStorageSync`。

- **Token 凭证**: 登录后的 JWT Token。
  - **存储位置**: `Storage` (持久化) + 内存。
  - **规则**: 每次应用启动时从 Storage 读取，网络请求拦截器中自动附加到 Header。过期或返回 401 时自动清空并触发重定向到登录。
- **当前用户信息 (`userInfo`)**: 包含当前登录用户的 id, nickname, avatar, 角色权限等。
  - **存储位置**: 全局状态 (内存) + `Storage` (持久化)。
  - **规则**: 在“我的”页面修改资料成功后，必须同步更新此全局状态，使得其他页面（如发布、评论时展示的当前头像）能立即响应。
- **系统级配置**: 如当前的设备信息（刘海屏高度、底部安全区）、网络状态等。

### 1.2 页面本地状态 (Local State)
> 存放在页面自身的 `data` (原生小程序) 或组件内部的 `useState` (React) 中。**严禁将以下数据放入全局状态**。

- **列表数据 (`list`)**: 首页瀑布流、需求列表、活动列表等。
- **表单输入 (`formData`)**: 发布作品、发布需求时的草稿数据。
- **UI 状态 (`isLoading`, `isModalVisible`)**: 加载中、弹窗显示隐藏等。

## 2. 核心场景数据流策略

### 2.1 列表刷新与分页策略 (首页、活动、需求)
- **初始化**: 进入页面触发 `onLoad`，拉取第一页数据。
- **下拉刷新 (`onPullDownRefresh`)**: 
  - 清空当前 `list`，重置 `page = 1`。
  - 重新请求 API，成功后停止下拉动画 `wx.stopPullDownRefresh()`。
- **上拉触底 (`onReachBottom`)**:
  - 检查是否还有更多数据 (`total > list.length` 或 `hasMore`)。
  - 如果有，`page += 1`，请求 API，将新数据 `concat` 拼接到现有 `list` 后。

### 2.2 发布流程的回流策略
**场景**: 用户在“首页” -> 点击“发布” -> 进入“发布页” -> 发布成功。
- **动作**: 发布成功后，调用 `wx.navigateBack({ delta: 1 })` 返回上一页。
- **数据流转**: 
  - **严禁方案**: 发布页把新数据存入全局状态，让首页去读。
  - **标准方案 (EventChannel 或 onShow 刷新)**: 
    - 方案A：发布页通过 `const eventChannel = this.getOpenerEventChannel(); eventChannel.emit('onPublishSuccess');` 通知首页。首页接收到事件后，执行下拉刷新的逻辑（重置第一页）。
    - 方案B：在首页的 `onShow` 生命周期中，判断是否需要静默刷新第一页数据（适合对实时性要求不极端的场景）。
    - **推荐方案 A (EventChannel)**，精确且性能好。

### 2.3 点赞/评论状态更新 (乐观更新)
**场景**: 用户在“帖子详情页”或“首页列表”点赞。
- **策略: 乐观更新 (Optimistic UI Update)**
  1. 用户点击“点赞”按钮。
  2. **立即**在本地页面状态中将该帖子的 `is_liked` 置为相反状态，并对应的 `like_count` +1 或 -1，触发 UI 立即变化。
  3. 发送 API 请求给后端。
  4. 如果后端返回成功，不做任何操作（因为 UI 已更新）。
  5. 如果后端返回失败（如网络错误），回退步骤 2 的本地状态修改，并 `Toast` 提示用户“操作失败”。
- **跨页面同步**: 详情页点赞后返回首页，首页的列表状态如何更新？
  - 详情页返回时，通过 `EventChannel` 传递 `{ postId: 1, is_liked: true, like_count: 121 }` 给首页，首页在自己的 `list` 中查找到对应的 `postId`，局部更新该项数据。

### 2.4 用户登录态与鉴权拦截
- **拦截器拦截**: 封装统一的 Request 方法。如果在需要鉴权的接口返回 `401 Unauthorized`：
  1. 暂停后续其他请求。
  2. 弹窗提示“登录已过期，请重新登录”。
  3. 跳转至登录页（或拉起微信授权弹窗）。
  4. 清除本地 Storage 中的 Token 和 UserInfo。
- **按钮级鉴权**: 
  - 对于“点赞”、“发布”、“报名”等核心操作按钮，点击时首先判断全局状态中是否有 Token。
  - 如果没有，拦截点击事件，直接拉起登录流程，而不是等 API 报错再处理。
